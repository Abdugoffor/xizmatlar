# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Stack

Laravel 12 (PHP 8.2+) admin + public site, Vite 7 with Tailwind v4, default DB is SQLite (`.env.example`). Frontend uses Blade only — no JS framework. Hosted under XAMPP on Windows; the repo path is `c:\xampp\htdocs\xizmatlar`.

## Commands

- `composer dev` — runs `php artisan serve`, `queue:listen`, `pail` (logs), and `npm run dev` together via concurrently.
- `composer test` — clears config then runs `php artisan test`.
- `composer setup` — bootstraps a fresh checkout (install, copy `.env`, key:generate, migrate, npm install, npm run build).
- `npm run dev` / `npm run build` — Vite dev server / production build (entries: `resources/css/app.css`, `resources/js/app.js`).
- `php artisan test --filter=<TestName>` — run a single test.
- `php artisan make:crud {ModelName}` — scaffold controller/requests/resource/views/routes from an existing `app/Models/{ModelName}.php`. Provided by the local-author package `abdugoffor/admin-crud-generator-with-api`. Companion commands: `make:apicrud`, `make:auth`, `make:apiauth`, plus a role/permission setup command. Most controllers + Blade views in this repo are generated by this tool — match its output style when editing.

## Architecture

### Locale-prefixed routing

All routes live inside `Route::prefix('{lang}')->middleware([LangMiddleware::class])` in [routes/web.php](routes/web.php). The bare `/` redirect is in [routes/lang.php](routes/lang.php) (included from `web.php`). Consequences:

- Every controller action takes `$lang` as the first route parameter, e.g. `show($lang, $id)`. Don't drop it.
- [app/Http/Middleware/LangMiddleware.php](app/Http/Middleware/LangMiddleware.php) resolves the active locale from URL → session → `config('app.locale')` against the active rows in the `languages` table, then calls `URL::defaults(['lang' => $lang])` so `route(...)` calls don't need to pass `lang` explicitly.
- Auth/admin areas are gated by the `role` middleware alias (registered in [bootstrap/app.php](bootstrap/app.php) → [app/Http/Middleware/RoleMiddleware.php](app/Http/Middleware/RoleMiddleware.php)). It checks `users.role` against an enum of `user`/`admin` defined inline on `User::$enumValues`.

### Translatable JSON fields

Multilingual content is stored as JSON columns on the model itself, not via a pivot. Every translatable model casts its translatable columns to `array` (see [app/Models/Service.php](app/Models/Service.php) for the canonical example: `title`, `description`, `content`, `footer_text`).

- Controllers always inject a `default` key from the first translation before saving: `$data['title']['default'] = reset($data['title'])`. Preserve this when editing store/update methods — `getLocale()` falls back to `default` when the requested locale is missing.
- Search filters use JSON path operators: `whereRaw("lower(title->>'{$locale}') LIKE lower(?)", ...)`. This works on Postgres and modern SQLite; if you switch DB engines, verify these queries.
- Validation rule generation: call `validateTranslation('column_name')` (in `app/helpers.php`) — it returns `["column.uz" => "required|string|max:5000000", ...]` for every active language.

### Global helpers (`app/helpers.php`)

This file is **required directly from [public/index.php](public/index.php)** — it is *not* in `composer.json` autoload. If you create a new helpers file, wire it the same way or it will silently not load. Key functions:

- `getLanguage()` — cached (`Cache::remember('getLanguage', 180min, ...)`) collection of active `Language` rows. When you mutate languages, `Cache::forget('getLanguage')`.
- `getTranslation($slug)` — DB-backed UI strings from the `translations` table, cached per-locale under `menyu.{slug}_{locale}`. [TranslationController](app/Http/Controllers/Translation/TranslationController.php) calls `forgetTranslationCache()` after writes; do the same anywhere you mutate translations.
- `getLocale($jsonArray)` — read the active locale from a JSON-cast attribute, falling back to `default`.
- `historyCheck(...$models)` — renders the audit-log Bootstrap modal HTML used in admin Blade views.
- `slug()` (custom Cyrillic→Latin transliterator) and Laravel's `Str::slug()` are both used; controllers default to `Str::slug()` against the `default` translation.
- `youtubeEmbed($url)` — normalize a YouTube URL to its embed form.

### Audit history (`HasHistory` trait)

[app/Traits/HasHistory.php](app/Traits/HasHistory.php) is mixed into nearly every model. It hooks `created/updated/deleted/restored` and writes to the `histories` table ([migration](database/migrations/2026_03_10_073709_create_histories_table.php)) with `model_type`, `model_id`, `language_code`, `old_data`, `new_data`. Notes for editors:

- The trait's update hook diffs JSON fields key-by-key — adding new translatable columns "just works" but make sure they're cast to `array`.
- If a model defines a `parent()` relation, history rows reattach to the parent (used for child translation rows).
- The diff renderer in `historyCheck()` hardcodes a language list `['uz','ru','en']` and special-cases keys named `photo`, `logo`, `qk_code`, `qk_code_path`, `passport_copy`, `is_active`, `requires_visa`, `status`. Add new file/enum keys here or they will render as plain strings.

### File uploads

[app/Services/FileUploadService.php](app/Services/FileUploadService.php) is the only upload path used by controllers. It:

- Writes to `public/uploaded/{images|videos|documents}/` (chosen by mime type), filename `dd-mm-YYYY-{30 random chars}.{ext}`.
- Returns the relative path string (`uploaded/images/...`) which is stored directly on the model.
- On error returns the exception message string instead of `null` — callers should not assume a clean null/path contract.

Models that own files expose `getFileFields(): array` (see `Service::$fileFields`). The CRUD generator uses this to decide which form inputs become file uploads.

### Layouts and view composers

Two Blade layouts:

- `layouts.admin` — back-office views under `resources/views/{aboutcompanies,services,blogs,...}`.
- `layouts.front` — public site under [resources/views/front/](resources/views/front/). [AppServiceProvider](app/Providers/AppServiceProvider.php) registers a `View::composer('layouts.front', ...)` that injects `$services`, `$logo` (latest `BannerPhoto`), and `$contact` into every front-end view — don't re-fetch these in controllers.

The public site is rendered by a single controller, [app/Http/Controllers/Front/IndexController.php](app/Http/Controllers/Front/IndexController.php), one method per page (`index`, `about`, `service`, `serviceShow`, `blog`, `blogShow`, `team`, `contact`).

### Admin CRUD pattern

Each admin entity has a parallel set of files; when adding a new one, mirror an existing example like `Service`:

- `app/Models/{Name}.php` (with `HasHistory`, JSON casts, optional `$fileFields`)
- `app/Http/Controllers/{Name}/{Name}Controller.php`
- `app/Http/Requests/{Name}/{Store|Update}{Name}Request.php` (use `validateTranslation()` for translatable cols)
- `app/Http/Resources/{Name}/{Name}Resource.php` (use `getLocale()` to flatten JSON)
- `resources/views/{names}/{index,create,edit,show}.blade.php`
- `Route::resource('{names}', {Name}Controller::class)` inside the `role:admin,user` group in [routes/web.php](routes/web.php)
